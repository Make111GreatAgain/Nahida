# Sum-check protocol

## 定义与范围

- 定义: Sum-check protocol 是交互式证明中的基础协议，用于证明多变量多项式在布尔超立方体上的求和结果。
- 不包含: 单篇论文、单个协议实例、一次实验结果或某个仓库模块的局部细节；这些留在 `03_Sources` source note 或本节点的 Source Extensions 行里。
- Canonical terms: `sum-check-protocol`
- Aliases/query keys: sumcheck
- Standalone completeness check: 本节点给出本地定义、边界、问题、方法族、代表 sources、缺口和更新记录；链接用于深入，不作为唯一解释。
- Retrieval role: 让查询优先从本节点理解 `sum-check-protocol`，再按需打开少量 source notes。
- Update scope: 新 source 若改变定义、方法族、代表 source、bridge 或 open problem，应刷新本节点。
- Domain dynamics note: not_applicable

## 背景

当前 source seed 包含 Thaler sum-check notes/PDF、Sparrow 的 space-efficient sumcheck extension、zkCNN 的 FFT/convolution sumcheck extension、Gemini 的 elastic scalar/tensor-product extension、Epistle 的 streaming multivariate PIOP extension，以及 zk-vSQL 的 committed sum-check extension。Sum-check 在 GKR/delegated computation 与后续 SNARK/VC 系统中作为基础构件；Sparrow 说明它可以被 specialized for streaming/low-memory prover settings，zkCNN 说明它也可以被 specialized for structured ML operator proving，Gemini 说明 sumcheck-style reductions 可被组织进 elastic R1CS PIOP，Epistle 说明 SumCheck/ZeroCheck/product/permutation checks 可以组织成 Plonkish/HyperPlonk streaming PIOP，zk-vSQL 则说明每轮 polynomial messages 可放入 homomorphic commitments 以隐藏 witness-dependent coefficients。
Public-coin recursion source adds a GKR/hash-verification use case: sum-check remains the layer-reduction engine inside GKR, while the reusable higher-level move is recursive verifier embedding into a SNARK.
Time-optimal GKR source adds a direct prover-efficiency case: for regular circuit layers, the sum-check polynomial and partial-evaluation arrays can be organized so total prover work over a layer is linear rather than one full layer scan per round.
HOBBIT adds a second product-form low-memory route: it keeps prover time optimal for arbitrary arithmetic-circuit SNARKs by combining a product-form streaming sumcheck with a transparent multilinear PCS and an execution-trace oracle.
