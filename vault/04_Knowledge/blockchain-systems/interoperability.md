# Blockchain interoperability

## 定义与范围

- 定义: Blockchain interoperability 研究多条区块链之间如何转移资产、交换资产、验证状态、传递消息或协同审计，并同时保持安全、隐私、效率和可组合性。
- 不包含: 单篇论文、单个 bridge/system 实例、一次实验结果或某个 smart contract 的局部细节；这些留在 `03_Sources` source note 或本节点的 Source Extensions 行里。
- Canonical terms: `blockchain interoperability`, `cross-chain interoperability`
- Aliases/query keys: cross-chain interoperability, 跨链互操作
- Standalone completeness check: 本节点给出本地定义、边界、问题、方法族、代表 sources、缺口和更新记录；链接用于深入，不作为唯一解释。
- Retrieval role: 让查询优先从本节点理解 blockchain interoperability，再按需打开 cross-chain protocols 和 source notes。
- Update scope: 新 source 若改变定义、方法族、代表 source、bridge 或 open problem，应刷新本节点。
- Domain dynamics note: [[04_Knowledge/blockchain-systems/research-dynamics|Research dynamics]]

## 背景

当前 source seeds 是 zkCross、zkBridge、sigBridge、DCCV、Scalable Cross-Chain Messaging、Atomic Commitment Across Blockchains 和 Cross-chain Deals，因此本节点仍是 foundation_thin，但已覆盖七类关键互操作问题。zkCross 说明互操作不只是“把资产从 A 链挪到 B 链”，还包括不同链之间的隐私保护、审计可验证性和审计效率。zkBridge 说明桥的 correctness trust boundary 同样关键: 小委员会签名能省 gas，但会把桥安全性降到 committee key security；light-client proof relay 则把正确性转给底层链和 proof soundness。sigBridge 说明在 permissioned blockchain 中，known membership 与 consensus signatures 可以成为桥验证材料，但要显式承担 trusted setup、signature scaling 和抽样验证概率边界。DCCV 则说明互操作还可能围绕可更新数据副本展开：当 source chain 上的动态数据发生授权更新时，target chain 的副本同步需要可审计的一致性验证。Scalable Cross-Chain Messaging 把视角推进到消息运输层：relayers 即使不被信任地验证消息，也会因为 permissionless redundancy、任务竞争和激励设计影响吞吐、公平性和抗审查。Atomic Commitment Across Blockchains 把视角推进到跨链原子性：多链资产交易需要共同的 redeem/refund decision，且这个 decision 的可信性、finality 和 evidence validation 是互操作方向的一等问题。Cross-chain Deals 则把跨链原子性推进到 adversarial commerce：复杂 deal 不只是双边 swap，还需要保护 compliant parties 的 acceptable payoff 和 escrow liveness。

FedChain 补充了 interoperability 的 security substrate 视角：SPV transfer 需要 source-chain finality 作为底座；在多链 PoS 环境中，stake/reward 分布不均会让某条链成为 federated system 的 weakest link。

MetaOpera 补充了 interoperability 的 application-object 视角：当互操作对象从 blockchain-native token/message 扩展到 metaverse avatar、virtual asset 和 object 时，协议不仅需要验证跨链事件，还需要处理 centralized server observation、NFT representation、object-format conversion 和平台 policy compatibility。它把 cross-chain technologies 与 oracle-based on-chain/off-chain exchange 组合成 relay-metaverse route。

Bridging BRC-20 to Ethereum 补充了 interoperability 的 heterogeneous-execution 视角：Bitcoin 的 UTXO/inscription 侧可以表达对 Ethereum smart contract 的调用意图，但中间必须有 middleware/validator committee 负责扫描、排序、共识、支付 gas 和回写 receipt。它把互操作从“资产或对象可迁移”推进到“源链 message/inscription 触发目标链 contract execution”，但当前证据仍是 one-way Bitcoin -> Ethereum route。

## 方法族与解决路线

| 方法/路线 | 核心思想 | 适用条件 | 代价/限制 | 代表来源 |
| --- | --- | --- | --- | --- |
| Chain-based protocols | 依赖链自身机制，如 sidechain、HTLC、SPV/Merkle proofs。 | 想避免强第三方或 relay chain 信任。 | 不同活动需要不同协议；隐私和审计能力常常不足。 | [[sha256-7ab37dce93ba-zkcross-cross-chain-privacy-preserving-auditing|zkCross]] |
| Bridge-based protocols | 用 notary、relay chain 或 supervision/audit chain 协调多链交互。 | 需要统一审计、消息转发或多链状态聚合。 | 引入第三方/中心化/单点风险，或需要额外共识与激励。 | [[sha256-7ab37dce93ba-zkcross-cross-chain-privacy-preserving-auditing|zkCross]] |
| Privacy-preserving auditing | 用 ZK proofs 将隐私交互与审计验证分离，让 auditors 验证 proof 而非读取完整交易。 | 跨链场景同时要求 unlinkability、auditability 和 full auditing efficiency。 | proving/setup/gas/匿名集合/denomination 都是系统边界。 | [[sha256-7ab37dce93ba-zkcross-cross-chain-privacy-preserving-auditing|zkCross]] |
| Trustless proof-based bridges | 用 sender-chain light-client transition proof 和 receiver-chain updater contract 替代 committee correctness trust。 | 目标链直接验证源链共识/签名过贵，且桥不想新增信任委员会。 | 仍需要处理 finality/orphan policy、relayer liveness、proof generation cost、application state proof 和 fee incentives。 | [[sha256-fbf50bc93d96-zkbridge-trustless-cross-chain-bridges|zkBridge]] |
| Permissioned signature-based bridges | 用 permissioned chain 已知 validator/consensus signatures 作为跨链 header verification evidence，目标链 updater contract 可全量或抽样验证签名。 | 两条链处于许可/联盟环境，源链 public keys 与 consensus rules 可安全初始化到目标链。 | signature count 增长会推高 gas/data size；抽样验证牺牲确定性；bridge liveness 仍需 honest relayer。 | [[sha256-9b3cfce6e018-sigbridge-cross-chain-bridge-permissioned-blockchains|sigBridge]] |
| Dynamic data consistency auditing | 用 source/target/audit chain 三方协作合约、periodic challenge 和 update-triggered proof 检查跨链数据传输和动态更新是否一致。 | 跨链交互对象是可更新数据，且存在监管/联盟链 audit chain 可以承担一致性 verifier。 | 依赖 honest audit chain 和 data owner；dynamic Merkle hash tree/decentralized chameleon hash 的实现与密钥治理复杂。 | [[doi-10-1016-j-jksuci-2023-101897-dccv-dynamic-cross-chain-data-consistency|DCCV]] |
| Permissionless relayer coordination | 用链上 Coordinator、registered relayer set、collateral、hash-based task allocation、delivery acknowledgement 和 timeout slashing 协调跨链消息运输。 | 多个 relayers 竞争同一消息导致 gas waste、吞吐不扩展或强 relayer 垄断时。 | 牺牲一部分冗余；依赖 timeout/proof-of-absence 和激励参数；复杂 QoS/负载均衡未解决。 | [[arxiv-2310-10016v1-towards-scalable-cross-chain-messaging|Scalable Cross-Chain Messaging]] |
| Atomic cross-chain transaction coordination | 用共同的链上可验证 commit/refund/abort decision 协调多条链上的资产锁定合约，并在 adversarial commerce 中保证 compliant parties 得到 acceptable payoff。 | 多链资产转移、交换、brokerage、auction 或 arbitrage 需要跨链原子性或 deal safety。 | HTLC/timelock 有 crash/delay/DoS 边界；witness/CBC/global-ledger route 需要 finality、evidence validation、censorship 和额外合约成本。 | [[atomic-cross-chain-transactions|Atomic cross-chain transactions]] |
| PoS-secured federated transfer | SPV proof 在 destination chain 上触发 token release，PoS consensus 和 dynamic rewards 保护 source-chain security 与 stake distribution。 | 多条 PoS/federated chains 之间转移 token，且 stake 可跨链流动。 | 不覆盖 arbitrary messaging/privacy；真实 staking market 和 chain governance 可能偏离 Stackelberg model。 | [[doi-10-1109-tsc-2023-3240235-fedchain-secure-proof-of-stake-framework-federated-blockchain-systems|FedChain]] |
| Cross-metaverse object interoperability | 用 relay metaverse 将源 metaverse 中锁定的 object 表示为 NFT，再通过 staking、committee/oracle proof 和 customizer format conversion 在目标 metaverse 中重铸。 | metaverse avatar、virtual asset 或 NFT-derived object 需要在 decentralized/centralized metaverses 之间迁移时。 | 委员会/Oracle trust、锁定/铸造接口、object data structure、platform policy 和 customizer 质量控制都仍是外部边界。 | [[arxiv-2302-01600-metaopera-cross-metaverse-interoperability|MetaOpera]] |
| Inscription-triggered contract invocation | Bitcoin inscription 携带目标 Ethereum contract address、operation 和参数；middleware validators 扫描 Bitcoin、排序请求、执行 Ethereum call，并把 receipt 再写回 Bitcoin。 | 源链缺少通用 smart-contract execution，但能通过 inscription/metadata 表达跨链调用意图时。 | 当前是 one-way Bitcoin -> Ethereum；依赖 validator committee honest majority、gas accounting、receipt inclusion 和 inscription 标准稳定性。 | [[arxiv-2310-10065-bridging-brc20-to-ethereum|Bridging BRC-20 to Ethereum]] |

## 当前综合

- Evidence window: `2026-06-20` to `2026-06-23`，仅覆盖当前 vault 已有 source。
- Freshness: `fresh` for current source absorption; not a latest-news claim.
- Valid until: `2026-07-23`。
- 综合: Blockchain interoperability 的当前 seed 指向 cross-chain protocols。zkCross 让本方向显式包含 privacy/auditability/full-auditing-efficiency 三角 tension；zkBridge 让本方向显式包含 bridge correctness trust boundary，即 committee trust、light-client verification cost、proof soundness 和 relayer liveness 之间的权衡；sigBridge 进一步说明在 permissioned setting 中，可以把已有 consensus signatures 作为 verification evidence，避免 ZK proving infrastructure，但要接受 signature verification 成本、trusted setup 和概率抽样边界。DCCV 新增的父级信号是 dynamic data consistency：跨链互操作有时需要验证 source-chain 上的授权更新是否被 target-chain 完整同步，这与 asset transfer/header relay 相邻但问题焦点不同。Scalable Cross-Chain Messaging 新增的父级信号是 transport coordination：跨链互操作不仅要证明消息正确，还要让 relayers 在 permissionless 环境中可扩展、可追责、不过度中心化地运输消息。Atomic Commitment Across Blockchains 新增的父级信号是 atomicity coordination：跨链资产 movement 需要共同的 commit/refund decision，且这不是 privacy、bridge verification 或 relayer transport 能自动解决的问题。Cross-chain Deals 进一步说明 atomicity coordination 不应只按简单 all-or-nothing swap 理解；当参与者自治且可恶意时，interoperability 还要显式处理 acceptable payoff safety、weak/strong liveness 和复杂 deal expressiveness。

- FedChain delta: interoperability 的一条新边界是 weakest-chain security。跨链合约可以验证 SPV proof，但 proof 的最终可信度来自 source-chain consensus 和 stake distribution。
- MetaOpera delta: interoperability 的一条新边界是 object semantics beyond chain state。跨链/Oracle 协议可以搬运 metaverse object ownership，但 object identity、format conversion、offline trustworthiness、universal data structure 和 universal policy 不能只靠 NFT wrapping 解决。
- BRC-20 bridge delta: interoperability 的一条新边界是 source-chain execution mismatch。Bitcoin inscription 可以携带 Ethereum 调用意图，但 invocation 需要中间层把 inscription message relaying、validator ordering、gas fee deduction 和 Ethereum receipt 写回串起来；这应路由到 [[cross-chain-message-relaying-to-cross-chain-smart-contract-invocation|Cross-chain message relaying -> cross-chain smart contract invocation]]，而不是 transaction processing。
